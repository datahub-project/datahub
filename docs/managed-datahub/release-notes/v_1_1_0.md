# v1.1.0

## Release Changelog

### v1.1.0

Includes all upstream OSS changes through DataHub Core [v1.6.0](https://github.com/datahub-project/datahub/releases/tag/v1.6.0) — see [Updating DataHub](https://datahubproject.io/docs/how/updating-datahub/) for breaking changes.

#### Release Availability Date

25-May-2026

#### Recommended Versions

- **CLI / SDK**: v1.5.0.19
- **Remote Executor**: v1.1.0-cloud
- **On-Prem Versions**:
  - **Helm**: TBD
  - **API Gateway**: TBD
  - **Actions**: TBD

#### New Feature Highlights

- **OAuth2 Dynamic Client Registration (RFC 7591)** — Third-party apps and MCP clients can now self-register via standard OAuth2 endpoints without manual setup.
- **CDE Governance MCP** — New MCP toolset exposing governance, fitness, lineage, and reliability capabilities to AI agents and IDE assistants.
- **DataHub Bridge** — New Kafka worker in Remote Executor that converts and forwards MCLs across all workloads, unifying executor-to-GMS flow on Kafka.
- **Ask DataHub follow-up suggestions** — The UI now suggests follow-up questions so users can explore answers without composing the next prompt themselves.
- **Java 25 LTS runtime** in official Docker images (Java 21 build toolchain).

#### Breaking Changes

The following items from upstream OSS v1.6.0 may require action. See [Updating DataHub — v1.6.0](https://datahubproject.io/docs/how/updating-datahub/#v160) for full details.

- **Spring Boot 4** — Custom GMS plugins or Spring extensions need recompile and retest. Standard Docker / Helm installs unaffected unless extending server classpath.
- **V1 UI removed** — V2 is required. Anyone still on V1 env flags must migrate.
- **Play 3 + `DATAHUB_SECRET`** — Secret must be ≥32 bytes or startup fails. Affects Compose and hand-crafted short secrets.
- **SQL Profiling Default** — Default profiler is now `sqlalchemy`. Recipes with `method: ge` need `acryl-datahub[profiling-ge]`.
- **Search `value` → `values`** — Singular `value` removed on filter Criterion. Custom REST, GraphQL, and SDK clients must use the `values` array.
- **Sigma** — Multiple lineage URN and field-path changes affect customers with warehouse-backed charts: chart `InputFields` self-references are dropped, column field paths now reflect warehouse-native names (not Sigma display names), workbook charts now emit warehouse-table lineage edges in `ChartInfo.inputs`, and a DM element schema bug fix may surface previously suppressed columns.
  - **Action — Redshift users:** Set `default_database` and `default_schema` per connection in the new `connection_to_platform_map` config, or lineage URNs will not match your warehouse connector. Snowflake and BigQuery users with default settings are unaffected. See [Updating DataHub OSS v1.6.0 docs](https://github.com/datahub-project/datahub/blob/v1.6.0/docs/how/updating-datahub.md#v160) for full detail.
- **Athena / Fivetran / Two-Tier SQL** — Upstream URN changes (Athena, Fivetran hybrid multi-destination); stored-procedure URN shape (MySQL, MariaDB, Hive, ClickHouse, Teradata). Lineage keyed on old URNs will break.
- **Unity Catalog — Ownership and Properties Now UPSERT by Default** — Each ingestion run fully replaces ownership and properties for each dataset; manually-added owners or properties in DataHub will be overwritten.
  - **Action:** To preserve manual edits, set `incremental_ownership: true` and / or `incremental_properties: true` in your recipe.
- **BigQuery Policy Tag Extraction Rewritten** — `extract_policy_tags_from_catalog` now uses `INFORMATION_SCHEMA` + batch Data Catalog API calls, eliminating millions of per-column API calls. Old code path removed with no fallback.
  - **Action:** If you use this feature, verify BigQuery API v2 is available; expect resource names instead of display names if the Data Catalog API is blocked by VPC Service Controls.
- **Micrometer / JMX** — Actuator on port **4319**; JMX scrape at **`/metrics`**. Prometheus and Grafana scrape configs need updating.
- **Actions / Kafka — Default Offset Commits Now Async** — Up to 25x throughput; on crash, ~10s of events may be redelivered. Built-in actions tolerate this.
  - **Action:** Non-idempotent custom actions should set `async_commit_enabled: false` under `source.config` to restore sync commits.
- **Auth** — `corpUserInfo.active` no longer considered for session eligibility.
- **Vertex AI** — Model version set URNs scoped per project. Orphaned version sets possible after upgrade.
- **dbt Assertion Result Types Now Distinguish Infra Errors from Data Failures** — Runtime errors (compile failures, missing sources, infra outages) now map to `type = ERROR` instead of `FAILURE`. New `severity` field on failures: `LOW` for `warn`, `HIGH` for `fail`.
  - **Action:** Alerts filtering on `type == FAILURE` will miss infra issues — include `type == ERROR` to keep capturing them.
- **Dataplex / Google Knowledge Catalog** — Two config fields renamed in `filter_config`. Recipes using the old names will fail.
  - `filter_config.entries.dataset_pattern` → `filter_config.entries.pattern`
  - `entries_location` (single) → `entries_locations` (list)
- **Docker Build** — `BASE_IMAGE` and `apkRepositoryUrl` build args required for custom image builds.

##### Context Hub

A new workspace for proposing, reviewing, and publishing AI-generated context (documentation, glossary terms, annotations) on assets — with human-in-the-loop governance.

- Detects common SQL patterns from query logs and proposes them as context documents with LLM-generated descriptions, related questions, and summarized logic — improving agentic text-to-SQL accuracy
- Inbox surfaces proposals with smart routing to likely domain experts for validation prior to publishing for agentic use
- Document history tracks lifecycle and annotation changes as business context evolves
- New MCP tools for agentic SQL workflows: `find_sql_context`, `draft_sql_for_tables`, `generate_sql_sketch`, and `note_metadata_observation`

##### Context Evals

Keep agent-facing context accurate and consistent. Evals continuously check context quality in DataHub and flag divergent or contradictory documentation before it's published.

- Create, edit, and delete eval questions from a dedicated Evals page
- Run evals individually or in groups against a dedicated eval-runner agent
- Surface the documents each eval referenced to debug answer quality
- Attach evals to context doc proposals to validate documentation changes before publishing

#### Ingestion — New Sources

- **Airbyte** — Workspaces, connections, sources, destinations, and streams, with asset- and column-level lineage from source → destination.
- **Aerospike** — Namespaces and sets with schema inferred from sampled records.
- **Apache Flink** — Jobs and run history with asset-level lineage from Kafka and SQL/Table API sources and sinks.
- **dltHub** — dlt pipelines and destination tables, with asset-level lineage end-to-end and column-level lineage for direct-copy sql_database pipelines.
- **Matillion DPC** — Projects, pipelines (orchestration, transformation, streaming/CDC), and execution history, with asset- and column-level lineage.
- **Microsoft Fabric Data Factory** — Pipelines, activities, and run history with asset-level lineage from Copy (source → sink) and ExecutePipeline (pipeline → pipeline) activities.
- **Omni** — Semantic models, topics, views, dashboards, and tiles with asset-level lineage along the Dashboard → Tile → Topic → View → physical table chain.
- **Pinecone** — Indexes and namespaces with vector-collection schema inferred by sampling.
- **StarRocks** — Catalogs (internal and external — Hive, Iceberg, etc.), databases, tables, and views.
- **Informatica Cloud (IDMC)** — Taskflows, Mapping Tasks, and source/target datasets with asset-level lineage.

#### DataHub Remote Executor

- **AWS Secrets Manager & GCP Secret Manager** — Native support for secret resolution.
- **DataHub Bridge** — New Kafka-based worker replacing SQSQ for moving metadata events from Remote Executor back to DataHub Cloud. Streamlines infrastructure dependencies (no AWS SQS / SNS required) and unifies MCL conversion across all workloads.
- **OAuth-Secured Kafka** — The executor now supports OAuth authentication when producing to Kafka, enabling integration with managed services (e.g., Confluent Cloud OAuth) and self-managed clusters using OIDC.
- **Faster, More Reliable Health Checks** — Health endpoint reduced to a sub-10ms liveness check. No longer fails during transient GMS outages, eliminating spurious pod restarts.
- **Proactive Background Task Monitoring** — Fetcher staleness, config resolver retries, scheduler jobs, and sweeper activity now emit Prometheus counters. Alerts fire within minutes of an issue, instead of waiting for retry exhaustion (up to 24h).

> **Operator note:** Use `WORKER_CONFIG_FETCHER_RETRIES` for alerting on sustained retries. The legacy `WORKER_CONFIG_FETCHER_ERRORS` gauge is preserved for backwards compatibility but no longer increments on every retry, so any existing alerts on it will silently miss issues.
