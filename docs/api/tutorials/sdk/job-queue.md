# Job Queue SDK

The Job Queue SDK provides distributed work claiming for DataHub services. It lets multiple worker instances discover, claim, and process work items without duplicates — using Compare-And-Swap (CAS) operations for atomic ownership.

**With the Job Queue SDK, you can:**

- Discover available work using search queries or event streams
- Atomically claim work so exactly one worker wins
- Release work when processing is complete
- Coordinate multiple worker instances without external infrastructure (no Redis, ZooKeeper, etc.)

## Getting Started

Install the DataHub Python SDK:

```bash
pip install acryl-datahub
```

Import from `datahub.sdk.patterns.job_queue`:

```python
from datahub.ingestion.graph.client import DataHubGraph, DataHubGraphConfig

from datahub.sdk.patterns.job_queue import (
    Claim,
    JobQueue,
    MCLDiscovery,
    SearchDiscovery,
    Sweeper,
)
```

Connect to your DataHub instance:

```python
graph = DataHubGraph(
    config=DataHubGraphConfig(server="http://localhost:8080")
)
```

## How It Works

The SDK has four layers:

```
┌──────────────────────────┐
│  JobQueue                │  ← Caller-facing API
│  poll() → List[Job]      │
│  Job.release()           │
├──────────────────────────┤
│  Discovery    │  Claim   │  ← Composed internally
│  poll() → URNs│  try/rel │
├───────────────┴──────────┤
│  ConditionalWriter       │  ← CAS mechanics
│  read_version()          │
│  write_if_version()      │
├──────────────────────────┤
│  DataHubGraph            │  ← Transport (existing)
└──────────────────────────┘
```

**Discovery** finds candidate entity URNs (via search queries or MCL event streams). **Claim** uses CAS to atomically take ownership by writing an aspect. **JobQueue** composes both into a poll-claim-process-release loop. You only interact with `JobQueue` — everything below is configuration.

The claim mechanism works by:

1. Reading the current aspect version from GMS
2. Checking whether the aspect is already in "claimed" state (via the `is_claimed` predicate)
3. Writing a "claimed" aspect value with an `If-Version-Match` header — GMS accepts only if the version hasn't changed since the read

If two workers race to claim the same entity, exactly one wins. The loser gets a 412 Precondition Failed response and `try_claim` returns `False`.

## Tutorial: Distributed Task Processing

This walkthrough uses PlatformResource entities as work items so you can follow along against any DataHub instance — no custom entities or aspects required.

**Convention used in this tutorial:**

| `platformResourceInfo` field | Purpose                                                             |
| ---------------------------- | ------------------------------------------------------------------- |
| `resourceType`               | Task state: `"TASK_PENDING"`, `"TASK_RUNNING"`, `"TASK_COMPLETED"`  |
| `primaryKey`                 | Resource-type marker (always `"task"`)                              |
| `value`                      | JSON blob with claim metadata: `{"owner": "...", "claimedAt": ...}` |

Each task is identified by its entity URN, which contains a GUID generated by whatever system creates the work (e.g., a scheduler, an API endpoint, or a user action). `resourceType` is indexed by DataHub search, so `SearchDiscovery` can find pending tasks efficiently.

### Step 1: Create Some Tasks

An external system creates PlatformResource entities for each unit of work, each with a unique GUID:

```python
import uuid

from datahub.emitter.mcp import MetadataChangeProposalWrapper
from datahub.ingestion.graph.client import DataHubGraph, DataHubGraphConfig
from datahub.metadata.schema_classes import PlatformResourceInfoClass

graph = DataHubGraph(
    config=DataHubGraphConfig(server="http://localhost:8080")
)

# Simulate an external system creating 3 pending tasks
task_urns = []
for _ in range(3):
    task_id = str(uuid.uuid4())
    task_urn = f"urn:li:platformResource:{task_id}"
    graph.emit(
        MetadataChangeProposalWrapper(
            entityUrn=task_urn,
            aspect=PlatformResourceInfoClass(
                resourceType="TASK_PENDING",
                primaryKey="task",
            ),
        )
    )
    task_urns.append(task_urn)
    print(f"Created {task_urn}")
```

### Step 2: Configure the JobQueue

Build a `JobQueue` that discovers pending tasks via search and claims them by CAS-writing the `platformResourceInfo` aspect. The owner ID and timestamps are stored in the `value` field as a JSON blob:

```python
import json
import os
import socket
import time

from datahub.metadata.schema_classes import SerializedValueClass

from datahub.sdk.patterns.job_queue import Claim, JobQueue, SearchDiscovery


def _now_ms() -> int:
    return int(time.time() * 1000)


instance_id = f"{socket.gethostname()}-{os.getpid()}"

queue = JobQueue(
    discovery=SearchDiscovery(
        graph=graph,
        entity_type="platformResource",
        filters={"resourceType": ["TASK_PENDING"]},
    ),
    claim=Claim(
        graph=graph,
        aspect_name="platformResourceInfo",
        aspect_class=PlatformResourceInfoClass,
        make_claimed=lambda owner, current: PlatformResourceInfoClass(
            resourceType="TASK_RUNNING",
            primaryKey=current.primaryKey if current else "task",
            value=SerializedValueClass(
                blob=json.dumps({"owner": owner, "claimedAt": _now_ms()}).encode(),
                contentType="JSON",
            ),
        ),
        make_released=lambda owner, current: PlatformResourceInfoClass(
            resourceType="TASK_COMPLETED",
            primaryKey=current.primaryKey if current else "task",
            value=SerializedValueClass(
                blob=json.dumps({"owner": owner, "completedAt": _now_ms()}).encode(),
                contentType="JSON",
            ),
        ),
        is_claimed=lambda aspect: aspect.resourceType == "TASK_RUNNING",
    ),
    owner_id=instance_id,
)
```

### Step 3: Process Tasks

Poll for pending tasks, claim them atomically, do work, and release:

```python
for job in queue.poll():
    try:
        print(f"Claimed {job.urn} — processing...")
        # ... do your work here ...
    finally:
        job.release()
        print(f"Released {job.urn}")
```

If you run this script from two terminals simultaneously, each task is claimed by exactly one worker — the other worker's CAS write fails and it moves on.

### Step 4: Verify

After processing, you can check that tasks moved to `TASK_COMPLETED`:

```python
for task_urn in task_urns:
    info = graph.get_aspect(entity_urn=task_urn, aspect_type=PlatformResourceInfoClass)
    if info:
        claim_data = json.loads(info.value.blob) if info.value else {}
        print(
            f"{task_urn}: state={info.resourceType}, "
            f"owner={claim_data.get('owner', 'none')}"
        )
```

## Discovery Strategies

### SearchDiscovery

Use `SearchDiscovery` when work items already exist and can be found via DataHub search filters. Good for startup catch-up, ownership-style work, and fallback recovery.

```python
discovery = SearchDiscovery(
    graph=graph,
    entity_type="platformResource",
    filters={"resourceType": ["TASK_PENDING"]},
    sort_field="requestedAt",  # optional: order results
    max_results=100,           # optional: limit per poll (default 100)
)

urns = discovery.poll()  # returns list of matching URNs
```

**Parameters:**

| Parameter     | Type                   | Description                                               |
| ------------- | ---------------------- | --------------------------------------------------------- |
| `graph`       | `DataHubGraph`         | DataHub graph client                                      |
| `entity_type` | `str`                  | Entity type to search (e.g., `"platformResource"`)        |
| `filters`     | `Dict[str, List[str]]` | Field-value filters (AND within fields, OR within values) |
| `sort_field`  | `Optional[str]`        | Field to sort results by                                  |
| `max_results` | `int`                  | Maximum results per poll (default: `100`)                 |

### MCLDiscovery

Use `MCLDiscovery` when work arrives as new entities or aspect changes and you need real-time event-driven discovery. Polls the DataHub MCL (Metadata Change Log) event stream.

:::note
MCLDiscovery requires a `DataHubEventsConsumer` implementation. Install `datahub-actions` to provide one, or pass a pre-built consumer via the `events_consumer` parameter.
:::

```python
discovery = MCLDiscovery(
    graph=graph,
    consumer_id=f"worker-{instance_id}",
    entity_type="platformResource",
    aspect_name="platformResourceInfo",
    is_candidate=lambda info: info.resourceType == "TASK_PENDING",
    batch_size=10,
    poll_timeout_s=5,
)

urns = discovery.poll()  # returns URNs from new MCL events
```

**Parameters:**

| Parameter         | Type                        | Description                                               |
| ----------------- | --------------------------- | --------------------------------------------------------- |
| `graph`           | `DataHubGraph`              | DataHub graph client                                      |
| `consumer_id`     | `str`                       | Unique identifier for this consumer instance              |
| `entity_type`     | `str`                       | Entity type to filter on                                  |
| `aspect_name`     | `str`                       | Aspect name to filter on                                  |
| `is_candidate`    | `Callable[[_Aspect], bool]` | Predicate to filter deserialized aspects                  |
| `topic`           | `str`                       | Kafka topic (default: `"MetadataChangeLog_Versioned_v1"`) |
| `batch_size`      | `int`                       | Max events per poll (default: `10`)                       |
| `poll_timeout_s`  | `int`                       | Long-poll timeout in seconds (default: `5`)               |
| `lookback_days`   | `int`                       | Days of history on first poll (default: `1`)              |
| `events_consumer` | `Optional[Any]`             | Pre-built events consumer to inject directly              |

## Configuring Claims

### Using Lambda Constructors

Lambda constructors give you full control over how the "claimed" and "released" aspect values are built. This is the approach used in the tutorial above, and works with any aspect structure — including nested fields like `PlatformResourceInfo.value`:

```python
claim = Claim(
    graph=graph,
    aspect_name="platformResourceInfo",
    aspect_class=PlatformResourceInfoClass,
    make_claimed=lambda owner, current: PlatformResourceInfoClass(
        resourceType="TASK_RUNNING",
        primaryKey=current.primaryKey if current else "task",
        value=SerializedValueClass(
            blob=json.dumps({"owner": owner, "claimedAt": _now_ms()}).encode(),
            contentType="JSON",
        ),
    ),
    make_released=lambda owner, current: PlatformResourceInfoClass(
        resourceType="TASK_COMPLETED",
        primaryKey=current.primaryKey if current else "task",
        value=SerializedValueClass(
            blob=json.dumps({"owner": owner, "completedAt": _now_ms()}).encode(),
            contentType="JSON",
        ),
    ),
    is_claimed=lambda aspect: aspect.resourceType == "TASK_RUNNING",
)
```

The `is_claimed` predicate is critical for correctness. It prevents sequential overwrites: without it, a second worker could read the latest version (written by the first worker's claim) and CAS-write over a legitimate claim. With `is_claimed`, the SDK reads the aspect value and returns `False` immediately if the entity is already claimed.

### Using `Claim.from_fields()` (Flat Aspects)

For aspects where the owner and state are simple top-level string fields, `from_fields` generates the `make_claimed` / `make_released` / `is_claimed` callables automatically:

```python
claim = Claim.from_fields(
    graph=graph,
    aspect_name="myStatusAspect",
    aspect_class=MyStatusAspectClass,
    owner_field="workerId",             # flat string field for the owner
    state_field="status",               # flat string field for the state
    claimed_state="RUNNING",
    released_state="COMPLETED",
    timestamp_field="startTimeMs",      # optional: auto-set to current time
    extra_claimed_fields={"report": ""}, # optional: extra fields on claim
)
```

This shortcut works well for domain-specific aspects with dedicated owner/state fields. It does not work when the owner is stored in a nested structure (like a JSON blob inside `PlatformResourceInfo.value`) — use lambda constructors for those cases.

**Parameters:**

| Parameter               | Type             | Description                                   |
| ----------------------- | ---------------- | --------------------------------------------- |
| `graph`                 | `DataHubGraph`   | DataHub graph client                          |
| `aspect_name`           | `str`            | Aspect used for claiming                      |
| `aspect_class`          | `Type[_Aspect]`  | The aspect's Python class                     |
| `owner_field`           | `str`            | Field name for the owner identifier           |
| `state_field`           | `str`            | Field name for the claim state                |
| `claimed_state`         | `str`            | State value representing "claimed"            |
| `released_state`        | `str`            | State value representing "released"           |
| `timestamp_field`       | `Optional[str]`  | Field to auto-populate with current time (ms) |
| `extra_claimed_fields`  | `Optional[Dict]` | Additional fields to set on claim             |
| `extra_released_fields` | `Optional[Dict]` | Additional fields to set on release           |

## Complete Examples

### Event-Driven Task Processing

Workers discover new tasks via MCL events as they arrive, claim them, process, and release. Good for online task queues where latency matters:

```python
import json
import os
import socket
import time

from datahub.ingestion.graph.client import DataHubGraph
from datahub.metadata.schema_classes import PlatformResourceInfoClass, SerializedValueClass

from datahub.sdk.patterns.job_queue import Claim, JobQueue, MCLDiscovery


def _now_ms() -> int:
    return int(time.time() * 1000)


instance_id = f"{socket.gethostname()}-{os.getpid()}"

queue = JobQueue(
    discovery=MCLDiscovery(
        graph=graph,
        consumer_id=f"worker-{instance_id}",
        entity_type="platformResource",
        aspect_name="platformResourceInfo",
        is_candidate=lambda info: info.resourceType == "TASK_PENDING",
        batch_size=10,
        poll_timeout_s=5,
    ),
    claim=Claim(
        graph=graph,
        aspect_name="platformResourceInfo",
        aspect_class=PlatformResourceInfoClass,
        make_claimed=lambda owner, current: PlatformResourceInfoClass(
            resourceType="TASK_RUNNING",
            primaryKey=current.primaryKey if current else "task",
            value=SerializedValueClass(
                blob=json.dumps({"owner": owner, "claimedAt": _now_ms()}).encode(),
                contentType="JSON",
            ),
        ),
        make_released=lambda owner, current: PlatformResourceInfoClass(
            resourceType="TASK_COMPLETED",
            primaryKey=current.primaryKey if current else "task",
            value=SerializedValueClass(
                blob=json.dumps({"owner": owner, "completedAt": _now_ms()}).encode(),
                contentType="JSON",
            ),
        ),
        is_claimed=lambda aspect: aspect.resourceType == "TASK_RUNNING",
    ),
    owner_id=instance_id,
)

# Worker loop
while running:
    for job in queue.poll():
        try:
            process_task(job.urn)
        finally:
            job.release()
```

### Search-Based Ownership at Startup

Workers use search to find unclaimed items at startup, claim them for the lifetime of the process, and release on shutdown. Good for long-running ownership (e.g., pipeline monitoring):

```python
queue = JobQueue(
    discovery=SearchDiscovery(
        graph=graph,
        entity_type="platformResource",
        filters={"resourceType": ["TASK_PENDING"]},
    ),
    claim=Claim(
        graph=graph,
        aspect_name="platformResourceInfo",
        aspect_class=PlatformResourceInfoClass,
        make_claimed=lambda owner, current: PlatformResourceInfoClass(
            resourceType="TASK_RUNNING",
            primaryKey=current.primaryKey if current else "task",
            value=SerializedValueClass(
                blob=json.dumps({"owner": owner, "claimedAt": _now_ms()}).encode(),
                contentType="JSON",
            ),
        ),
        make_released=lambda owner, current: PlatformResourceInfoClass(
            resourceType="TASK_COMPLETED",
            primaryKey=current.primaryKey if current else "task",
            value=SerializedValueClass(
                blob=json.dumps({"owner": owner, "completedAt": _now_ms()}).encode(),
                contentType="JSON",
            ),
        ),
        is_claimed=lambda aspect: aspect.resourceType == "TASK_RUNNING",
    ),
    owner_id=instance_id,
)

# Startup: claim available work
for job in queue.poll():
    start_monitoring(job.urn)

# ... run until shutdown signal ...

# Shutdown: release all
for job in queue.held_jobs:
    stop_monitoring(job.urn)
    job.release()
```

## Best Practices

- **Use a unique `owner_id` per worker instance.** Combine hostname + PID or use a UUID — this ensures claims are distinguishable across workers.
- **Always release in a `finally` block.** This prevents leaked claims if processing raises an exception.
- **Call `release_all()` on shutdown.** Register it as an `atexit` handler or in your shutdown hook to clean up held claims.
- **Heartbeat/expiry is the caller's responsibility.** The SDK does not manage lease timeouts. If you need heartbeats, delay calling `release()`. If you need expiry, run a Sweeper (see below).
- **Use one thread per entity for claim operations.** The CAS mechanism is not thread-safe for concurrent claims on the same URN from the same process.

## Crash Recovery with the Sweeper

Workers can crash without releasing their claims. The `Sweeper` periodically finds stale claims and force-releases them via CAS, so other workers can re-claim the work.

```
Worker-1 claims task     Worker-1 crashes     Sweeper runs          Worker-2 re-claims
─────────────────────    ────────────────     ─────────────────     ──────────────────
status = RUNNING         (no release)         status = TIMED_OUT    status = RUNNING
owner  = worker-1                             (force CAS write)     owner  = worker-2
```

### Using `Sweeper.from_fields()`

For flat aspects with dedicated state/timestamp fields, `from_fields` builds the sweeper with sensible defaults — including a `SearchDiscovery` that finds entities in the claimed state:

```python
from datahub.sdk.patterns.job_queue import Sweeper

sweeper = Sweeper.from_fields(
    graph=graph,
    aspect_name="myStatusAspect",
    aspect_class=MyStatusAspectClass,
    entity_type="myEntityType",
    owner_field="workerId",
    state_field="status",
    claimed_state="RUNNING",
    swept_state="TIMED_OUT",
    timestamp_field="startTimeMs",
    timeout_ms=3600_000,  # claims older than 1 hour are stale
)
```

### Using Lambda Constructors

For full control over the staleness check and the swept aspect value:

```python
from datahub.sdk.patterns.job_queue import SearchDiscovery, Sweeper

sweeper = Sweeper(
    graph=graph,
    aspect_name="platformResourceInfo",
    aspect_class=PlatformResourceInfoClass,
    is_stale=lambda aspect: (
        aspect.resourceType == "TASK_RUNNING"
        and _age_ms(aspect) > 3600_000
    ),
    make_swept=lambda aspect: PlatformResourceInfoClass(
        resourceType="TASK_TIMED_OUT",
        primaryKey=aspect.primaryKey,
    ),
    discovery=SearchDiscovery(
        graph=graph,
        entity_type="platformResource",
        filters={"resourceType": ["TASK_RUNNING"]},
    ),
)
```

### Running the Sweep Loop

Run the sweeper on a dedicated master process. Only one instance should run at a time — distributed leader election is the caller's responsibility.

```python
import time

while running:
    result = sweeper.sweep()
    logger.info(
        "Swept %d, failed %d, skipped %d",
        len(result.swept), len(result.failed), len(result.skipped),
    )
    time.sleep(600)  # every 10 minutes
```

`sweep()` returns a `SweepResult` with three lists:

| Field     | Description                                                      |
| --------- | ---------------------------------------------------------------- |
| `swept`   | URNs successfully force-released                                 |
| `failed`  | URNs where the CAS write failed (entity was released or changed) |
| `skipped` | URNs that weren't stale after reading the current aspect         |

CAS failures in `failed` are expected and harmless — they mean the entity was released or re-claimed between the sweeper's read and write.

### `Sweeper.from_fields()` Parameters

| Parameter            | Type             | Description                                     |
| -------------------- | ---------------- | ----------------------------------------------- |
| `graph`              | `DataHubGraph`   | DataHub graph client                            |
| `aspect_name`        | `str`            | The claim aspect to sweep                       |
| `aspect_class`       | `Type[_Aspect]`  | The aspect's Python class                       |
| `entity_type`        | `str`            | Entity type for SearchDiscovery                 |
| `owner_field`        | `str`            | Field containing the claim owner ID             |
| `state_field`        | `str`            | Field indicating claim state                    |
| `claimed_state`      | `str`            | Value of state_field when claimed               |
| `swept_state`        | `str`            | Value to set on force-release                   |
| `timestamp_field`    | `str`            | Field containing claim timestamp (epoch ms)     |
| `timeout_ms`         | `int`            | Claims older than this are considered stale     |
| `extra_swept_fields` | `Optional[Dict]` | Additional fields to set on sweep               |
| `max_results`        | `int`            | Max candidates per sweep cycle (default: `100`) |

## API Reference

### Core Classes

| Class             | Description                                                                              |
| ----------------- | ---------------------------------------------------------------------------------------- |
| `JobQueue`        | Top-level API. Composes Discovery + Claim into poll-claim-release.                       |
| `Job`             | Handle to a claimed work item. Has `urn` attribute and `release()` method.               |
| `Claim`           | Atomic claim/release via CAS. Tracks held versions internally.                           |
| `Sweeper`         | Periodic cleanup of stale claims. Uses Discovery + CAS to force-release.                 |
| `SweepResult`     | Result of `sweep()`: lists of `swept`, `failed`, and `skipped` URNs.                     |
| `Discovery`       | Protocol for work-discovery strategies. Implementations must have `poll() -> List[str]`. |
| `SearchDiscovery` | Discovers work via DataHub search queries.                                               |
| `MCLDiscovery`    | Discovers work by polling the MCL event stream.                                          |

### Shared Utilities (Advanced)

| Class               | Description                                                                        |
| ------------------- | ---------------------------------------------------------------------------------- |
| `ConditionalWriter` | Low-level CAS operations: `read_version()`, `read_aspect()`, `write_if_version()`. |
| `CASResult`         | Dataclass returned by `write_if_version()`: `success`, `new_version`, `reason`.    |

These are in `datahub.sdk.patterns._shared` and generally only needed if building custom claim logic outside of the `Claim` class.
